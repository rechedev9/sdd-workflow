## Decision

Every phase assembler injects a compact (~500–800 byte) cumulative summary of completed prior phases into its context. This summary is built by extracting the first heading + N lines from each prior artifact (`exploration.md`, `proposal.md`, `design.md`, review verdict), not by re-loading the full artifacts.

## Context

Downstream phases (e.g., `tasks`, `apply`, `review`) benefit from knowing the key decisions made in earlier phases. But loading full prior artifacts (proposal, design, spec) into every phase context would quickly hit the 100KB size guard and burn tokens unnecessarily.

The cascade trades completeness for conciseness: it captures the decision headings and first meaningful lines — enough to orient the LLM on the change intent and key choices — without the full prose.

## Alternatives Considered

- **Include full prior artifacts in every phase context** — rejected because it bloats context by 10–30KB per phase and risks hitting the 100KB size guard.
- **No prior context in downstream phases** — rejected because `tasks` and `apply` would lose the design rationale and generate inconsistent implementations.
- **Structured summary generated by an LLM call** — rejected because it adds a token cost and a failure mode for every context assembly.
- **Per-phase explicit artifact references (Claude reads files on demand)** — considered as an alternative; rejected for now because it requires Claude to make additional tool calls, increasing latency.

## Consequences

**Positive:**
- Downstream phases have the key context without re-paying the full artifact cost.
- Summary is deterministic (line-scanner, no LLM involved) and fast.
- 100KB size guard is rarely hit in practice.

**Negative:**
- Summary is shallow — only first headings and N lines per artifact. Nuanced mid-document decisions are not captured.
- If an artifact's most important content is not in the first heading block, the cascade misses it.
